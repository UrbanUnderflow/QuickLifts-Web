# Refresh active variant_a experiment results before making funnel partner messaging

## Core narrative

Funnel-partner messaging must be based on the current active `variant_a` experiment readout, not stale assumptions. Before any partner-facing funnel copy, campaign framing, or trial-start recommendation is approved, the active `variant_a` result set needs to be refreshed and read in its current state. If the readout is old, partial, or inferred from prior decisions, the message is not ready.

## Messaging pillars

### 1. Scoreboard first, story second

The Scoreboard and current Experiments data set the boundary for what we are allowed to say. If refreshed active `variant_a` performance is holding, messaging can reinforce the winning trial-start story. If it is slipping, we do not decorate the problem with stronger copy. We change the message only after the live read tells us what is still true.

### 2. Purchase behavior and drop-off tell us what to clarify

Purchase logs, cancel reasons, user state, and retargeting state turn a result into an explanation. They help us see whether active `variant_a` is winning because the offer is clear, stalling because trust drops after the first screen, or underperforming because follow-up is mismatched to where the user actually is. Messaging gets better when it answers the friction the operating data is already showing us.

### 3. Source quality changes the message we can responsibly scale

AppsFlyer imports help separate source quality from message quality. If active `variant_a` looks strong but only for certain acquisition sources, we should not stretch that result into a universal funnel claim. We pair AppsFlyer source reads with Scoreboard, Experiments, and purchase logs so the next message reflects who is converting well, who is hesitating, and which story is safe to scale without breaking trust.

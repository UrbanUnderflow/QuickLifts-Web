## Core narrative

Partner-facing funnel messaging must be updated only after the active `variant_a` experiment results are refreshed from the live operating inputs. That means we do not move forward on partner-facing funnel copy, campaign framing, or trial-start messaging from stale assumptions, prior narratives, or outdated snapshots. ## Messaging pillars

### 1. Scoreboard and Experiments define the safe claim

The Scoreboard and the live Experiments readout are the first boundary for partner-facing messaging. If refreshed active `variant_a` performance is holding, the message can reinforce the current funnel story. If it is slipping, we do not compensate with louder copy. We only say what the current Scoreboard and Experiments state can honestly support.

### 2. Purchase logs, cancel reasons, user state, and retargeting state explain the friction

Purchase logs show whether the funnel is converting, while cancel reasons, user state, and retargeting state explain where the trust gap may be opening. Together, those inputs tell us whether active `variant_a` is winning because the offer is clear, slowing because follow-up is mismatched, or leaking because the first step is asking for too much trust. Messaging should respond to that real friction, not an invented one.

### 3. AppsFlyer imports tell us what can scale without distortion

AppsFlyer imports help separate message performance from source quality. If refreshed active `variant_a` results are strong only for certain acquisition sources, we should not stretch that outcome into a broad partner claim. Source-aware reading keeps the funnel story precise and shows which message is safe to scale versus which one only works under a narrower mix of traffic.

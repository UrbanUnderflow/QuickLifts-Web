# Refresh Active Variant A Experiment Results Before Making Funnel Partner Messaging — 2026-06-25

## Core Narrative

Before any funnel-partner messaging change is approved, the active `variant_a` experiment results must be refreshed against the current operating inputs. That means we do not rewrite partner-facing messaging, adjust trial-start framing, or recommend a new funnel move from stale reads, yesterday's assumptions, or partial screenshots. We refresh the live `variant_a` result set first, then decide. If the active results are not current, the message is not ready.

## Messaging Pillars

### 1. Refresh Before Reframe

We anchor every messaging decision in the newest read across Scoreboard and Experiments first, then pressure-test it against purchase logs. If active `variant_a` performance has moved, partner messaging has to move from that fresh reality — not from the story we told last week. This keeps trial-start copy honest because the promise is tied to current conversion behavior, not a stale win rate.

### 2. Trust the Drop-Off, Not the Hunch

When refreshed `variant_a` results suggest a shift, we validate the why with cancel reasons, user state, and retargeting state before rewriting the funnel story. That protects us from overreacting to surface-level conversion changes and helps us distinguish between a real trust gap, an onboarding mismatch, and a retargeting follow-up problem. Messaging earns trust when it answers the friction users are actually showing us.

### 3. Source Quality Shapes the Story

AppsFlyer imports and purchase logs tell us whether `variant_a` is performing the same across acquisition sources or whether source quality is distorting the headline result. We use that source-aware read alongside Scoreboard and Experiments so partner messaging stays precise: the right message for the right audience, not one broad claim stretched across users who are entering with different intent and different likelihood to start a trusted trial.

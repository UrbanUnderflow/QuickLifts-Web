# Refresh active variant_a experiment results before making funnel partner messaging

## Core narrative

Funnel-partner messaging must only move forward after the active `variant_a` experiment results have been refreshed against the current Scoreboard and Experiments state. We do not approve partner-facing funnel copy, campaign framing, or trial-start recommendations from stale assumptions, old screenshots, or prior narratives that have not been checked against the live operating readout. If the active `variant_a` result has not been refreshed against the current Scoreboard and Experiments view, the message is not ready to move.

## Messaging pillars

### 1. Scoreboard sets the boundary

The Scoreboard is the first guardrail for what we are allowed to say. If refreshed active `variant_a` results are holding, messaging can reinforce the current funnel story. If they are slipping, we do not compensate with louder language. The message must stay inside what the live Scoreboard and experiment state can actually support.

### 2. Purchase and friction data explain the story

Purchase logs, cancel reasons, user state, and retargeting state tell us why a refreshed `variant_a` result is moving. They show whether the current message is earning trust, losing people after the first step, or mismatching the follow-up to where the user actually is. Messaging should answer the friction the operating data is surfacing, not the friction we imagine is there.

### 3. Source quality decides what can scale

AppsFlyer imports help us separate message strength from source quality. If active `variant_a` looks healthy only for part of the acquisition mix, we should not stretch that result into a broad partner claim. Source-aware reading keeps the funnel story specific, protects trust, and tells us which message is safe to scale versus which one only works under a narrower set of inputs.
